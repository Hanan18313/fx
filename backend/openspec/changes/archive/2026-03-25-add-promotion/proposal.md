## Why

系统已具备邀请码和上下级关系（`users.invite_code` / `users.parent_id`），但目前注册绑定后没有任何后续激励。用户没有动力去推广，推荐关系形同虚设。需要补全推广闭环：**邀请注册得奖励 + 分享商品赚佣金**，让推荐关系真正产生价值，驱动用户自发裂变。

## What Changes

- **邀请注册奖励**：新用户通过邀请码注册成功后，推荐人获得固定金额奖励，直接入账钱包余额
- **分享订单佣金**：用户分享商品链接，被分享者通过该链接下单（订单完成后），分享者获得订单利润池的一定比例作为佣金
- **推广记录追踪**：新增推广奖励流水表，记录每笔邀请奖励和分享佣金的来源、金额、时间
- **推广数据看板**：C 端用户可查看自己的推广统计（邀请人数、累计佣金、下级列表）
- **Admin 推广管理**：管理后台可查看推广数据概览、调整奖励参数

## Capabilities

### New Capabilities
- `referral-reward`: 邀请注册奖励机制——新用户注册绑定推荐人后，推荐人自动获得奖励入账
- `share-commission`: 分享订单佣金机制——被推荐人下单完成后，推荐人获得佣金
- `promotion-stats`: 推广数据查询——C 端用户查看推广统计和下级列表
- `promotion-admin`: Admin 端推广管理——查看推广数据概览、配置奖励参数

### Modified Capabilities
（无需修改现有规格——注册流程和订单流程的核心行为不变，推广逻辑以新增服务/事件的方式接入，不改变原有 API 契约）

## Impact

- **数据库**：新增 `promotion_rewards` 表记录推广奖励流水；新增 `promotion_configs` 表存储可配置的奖励参数
- **现有代码集成点**：
  - `AuthService.register()` — 注册成功后触发邀请奖励发放
  - `OrderService.payOrder()` — 订单完成后触发分享佣金计算
  - `WalletService` — 奖励和佣金入账复用现有钱包余额机制
- **新增模块**：`promotion` 模块（C 端 API）、Admin `promotion-mgmt` 模块
- **API 新增**：约 5-6 个新接口（推广统计、下级列表、Admin 数据概览、Admin 参数配置等）
- **依赖**：无新外部依赖，复用现有 TypeORM + Redis 基础设施
